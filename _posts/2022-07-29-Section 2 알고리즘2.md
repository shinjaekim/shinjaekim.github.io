## 정리
    프로그래밍 언어의 문법을 정확히 알아라
    조건에 부합하도록 로직을 짜라
    빠르게 작성하자.

구현이라 함은 무언가 생각한 바로 유의미한 결과를 도출하는 것이라고 생각한다. 데이터를 **정렬** 하고, **탐색**하는 등의 **문제를 보면 어떻게 구현할지**를 생각해 봐야할 것이다.

    완전탐색 - 가능한 경우의 수 전부 확인
    시뮬레이션 - 복잡한 구현 요구 사항을 바탕으로 시뮬레이션

### 1. 시뮬레이션
예시는 내가 직접 만들어서 해보고 싶다. 일단 먼저 시뮬레이션을 접하게 되면 중요한 것은 **독해** 능력이다. 로직을 설명해주기 때문에 로직을 정확히 이해하고 그에 맞는 코드를 하나하나씩 작성해가면 될 것이다. 코드가 길어진다고 해도 포기하지말자.

### 2. 완전 탐색 알고리즘
    Brute Force - 시행착오 방법론

***될때까지 한다***를 떠올리자. 프라하에서 공부할때 한번은 쇼핑몰 라커에 짐을 보관하고 비밀번호를 까먹은 적이 있었다. 물론 어느정도 내가 평소에 사용하던 값들이 있었기 때문에 0000부터 9999까지 다 해보지는 않았지만 될때까지 내가 사용했으리라 추측되는 비밀번호들을 하나씩 다 입력했었다. 이게 그 예시가 될 것 같다. (추측해서 추려냈다는 것은 엄격히 말하면 지적 능력을 사용했기 때문에 완전탐색은 아니다. 그래도 하나씩 대입해보며 해봤다는 것을 떠올리자.)

이런 이유로 완전 탐색 알고리즘은 성능이 좋은 컴퓨터에서 사용해야 할 것이다. 시간복잡도에 대한 계산 없이 일단 때려박는 무식한 방법이기 때문이다. ***딱히 좋은 다른 알고리즘이 없을때***, ***여러가지 경우의 수들을 비교해볼때*** 사용하자. 

후자 ***여러가지 경우의 수들을 비교해볼때*** 라는 뜻은 추상적으로 다가올 수 있으니까, 미래의 나를 위해 예시를 추가해보고 싶다.

    서울을 가야한다.
    국도, 고속도로 / 기차 / 비행기 등의 여러 수단들이 있다.

    그렇다면

    가장 빠른 길을 완전탐색 식으로 찾으면 하나씩 다 이용해보는 것이다.

글을 작성하다가 생각해보니 위의 내 경험도 이를 설명하기 위한 예시가 될 수 있겠다. 내가 가지고 있는(추측한) 비밀번호들을 하나씩 대입해 본 것은 여러가지 경우의 수를 확인해 본 셈이기도 하다.

어쨌든 이런 경우를 제외하면 굉장히 비효율적이다. 그러니 가능하다면 다른 알고리즘을 이용하자.

#### a. 이용 예시
- 순차 검색 알고리즘
- 문자열 매칭 알고리즘
- 선택 정렬 알고리즘
    - 0번 인덱스를 가리키고 시작
    - 1번 인덱스부터 끝까지 최소값을 구하고
    - 최소값이 현재 가리키는 값도가 크다면
    - 현재 인덱스로 옮긴다.
- 버블 정렬 알고리즘
- Tree의 완전탐색 : BFS DFS
- 동적 프로그래밍

### 3. 이진 탐색 알고리즘
    분할 정복 알고리즘 (절반씩 분할)

값들이 정렬되어 있다면 처음부터 끝까지 하나하나 찾기보다는, 중간값과 비교해서 한쪽을 버리는 것이 빠를 것이다. 이진 탐색 알고리즘의 다른 이름이 분할정복 알고리즘인 이유이기도 하다.

시간복잡도는 logn을 가진다. ***왜냐 계속 반씩 나뉘니까***

하지만 좋은건 상황이 제약적이다. 앞서 말했든 ***정렬되어 있어야***하고, ***배열에만 구현***할 수 있다.

- 더 찾아볼 것
    - 이진 트리와 이진 탐색 트리

#### a. 외의 탐색 알고리즘
- 선형 탐색 알고리즘
    - 때에 따라서 (데이터 양이 적거나 찾을값이 앞에 있는 경우) 이진 탐색보다 빠르기도 하다.
- 해쉬 탐색 알고리즘
- divide and conquer (정렬임)
    - 더 이상 쪼개질 수 없을 때 까지 쪼개고 조건에 맞게 정렬 한 이후 다시 합치는 과정을 가진다.
