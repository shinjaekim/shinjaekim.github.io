## 정리

### 1. 웹
    인터넷에서 제공하는 하이퍼텍스트 시스템

앞서 네트워크에서 정리했듯이, 백엔드 개발자로서의 성장과정에서 웹 애플리케이션을 떼놓을 수가 없다.

TCP/IP 통신 프로토콜과 함께 웹을 떠올려보면..    
WWW는 통신망 프로토콜과 데이터전달 프로토콜을 기반으로 한 정보 공유를 위한 구현물(공간)이다.

지금까지의 이해를 심화시켜보면 사실상 데이터 전달을 위한 프로토콜들은 기능들의 일부일뿐 TCP/IP 4단계에서 응용계층에 속하는 것들이다. (HTTP, FTP 등등)

그렇다면.. 웹을 네트워크 관점에서 더 나아가 구조 관점에서 파악해보자.

#### a. 클라이언트 서버 아키텍쳐

    클라이언트 <----------> 서버

웹 서비스는 큰 틀에서 서비스를 소모하는 '클라이언트'와 서비스를 제공하는 '서버'로 나뉜다. 브라우저를 통해 넷플릭스를 이용하는 상황을 가정해보자. 보고싶은 드라마를 고르고, 예능들을 둘러보고, 영화를 보는 등의 서비스를 소모하는 우리는 클라이언트 입장에 있는 것이다. 지금까지의 내가 이래왔다.

하지만 백엔드 개발자로서 해야하는 일은 클라이언트보다 서버쪽에 가깝다. 아직은 추상적이지만 클라이언트에서 요청한 일들에 대하여 여러 비지니스 로직들을 통해 알맞은 데이터로 매치시켜주는 것.. 이러한 일들이 서버에서 이루어 질 것이다.

위 문단을 적용하면 2티어 아키텍쳐를 3티어로 표현할 수 있다.

    클라이언트 <----> 서버 <----> 데이터베이스
    
    리소스 사용       제공           저장

데이터베이스의 저장과 제공이 분리된 것이다.

#### b. 구분하는 이유
일단 기능상으로 현실세계에서는 위와 같이 구현이 되어있다. 그래서 구분한다.

하지만 결과론적인 이야기가 아니라 원인에 대해서 생각해보자.

내가 경기장 예매 서비스를 지원하는 웹 애플리케이션을 만들었다고 가정해보자. 그 상황에서

    서버와 클라이언트 구분 x

위와 같은 설계를 했다면??? 경기장 좌석 하나 하나가 예매될때마다 어플리케이션이 데이터를 포함해서 계속 새로운 버전으로 업데이트 되어야 할 것이다. 교환학생때 기숙사 옆 스트라호브 경기장이 20만명을 수용할 수 있었다. 그러면 매 주 경기가 있다고 가정했을때 주마다 20만번의 어플리케이션 업데이트를 해주어야 한다.

따라서 서비스의 사용과 데이터의 제공을 분리시켜 놓은 것이다.

### 2. 웹 애플리케이션 아키텍쳐
웹 애플리케이션은
    신뢰성 - 확실히 의도한 바를 해줘야하고 
    확장성 - 확장이 자유로워야하며 
    보안성 - 정보를 안전하게 하고
    견고성 - 구조가 무너지지 않아야한다???

    (내 나름대로 해석한 것)

이렇게 4개의 요소를 충족하는 아키텍쳐를 가져야 한다.

그렇다면 아키텍쳐는 무엇인가??

    구분된 기능들이 어떻게 구현되는가??

    더 풀어서

    웹 브라우져에서 클라이언트가 요청을 하면,,
    웹 애플리케이션의 요소들이 어떻게 상호작용을 하는가??

라고 정리하자.

#### a. 웹 애플리케이션 요청흐름

    1. 브라우저에 도메인 주소를 입력한다.
    2. 도메인 주소로 부터 IP주소를 얻기 위해 DNS질의를 하게 된다. 포트 53
        리졸버가 root에게 물어보고
        TLD네임 서버에 물어보고
        권한있는 네임서버에 물어 본 후 도메인으로부터 IP주소를 얻게 된다.
    3. 해당 IP주소로 웹 서버에 접근한다.
        1,2,3 -> 웹 서버 접근

    4. 웹 애플리케이션을 조작하며 여러 요청을 보낸다.
    5. 서버에서 클라이언트의 요청으로부터 비즈니스 로직을 실행한다.
    6. 비즈니스 로직에 상응하는 데이터 가져온다.
    7. 브라우저에 응답한다.
        4,5,6,7 -> 웹 서버와의 상호작용

여기서 5번을 더 미시적으로 접근하도록 하자.

### 3. HTTP
그러면

위 요청 흐름에서 **클라이언트 -> 서버** 로의 요청은 어떠한 방식으로 이루어질까?

이를 구현해주는 기술이 HTTP이다.

    HTTP - HyperText Transfer Protocol

    무상태성이 특징

#### a. HTTP messages

    클라이언트 --------> 서버 : HTTP requests
    클라이언트 <-------- 서버 : HTTP responses

    -----------------------------
    start line (status line - responses)
    -----------------------------
    H
    T
    T
    P
    H
    E
    A
    D
    E
    R                             여기까지가 메세지의 '헤드'
    ㅁㅁㅁㅁㅁㅁ empty line ㅁㅁㅁㅁㅁㅁ -------------------
    body                          여기부터가 payload
    -----------------------------

- start line : 요청이나 응답의 상태, 항상 첫 줄
- HTTP headers : 요청 지정 or 본문 설명하는 헤더의 집합
- empty line : 헤더와 본문 구분
- body : 데이터나, 관련된 문서

#### b. request

    메서드, 요청 URL, HTTP 버전
    GET /document/item/1 1.1

Start line

    1. 수행할 작업에 대한 명령어(get, put, post...)등과 같은 HTTP method
    2. 요청대상
        - origin 형식
            ?와 쿼리 문자열

        - absolute 형식
            완전한 URL형식
            GET url HTTP/1.1

        - authority 형식
            HTTP터널을 구축하는 경우
            도메인 이름과 포트번호로 이루어짐
            CONNECT 도메인 : 포트번호 HTTP/1.1

        - asterisk 형식
            OPTIONS * HTTP/1.1

Headers

    헤더 이름 : 값
    1. Request headers
        - 가져올 리소스나 클라이언트 자체에 대한 정보
        - 구체화 헤더
            - USER-AGENT
            - ACCEPT-TYPE
            - ACCEPT-LANGUAGE
    2. General headers
        - 메시지 전체에 적용되는 헤더
        - body와는 관련이 업습니다.
    3. Representation headers
        - body에 담긴 리소스 정보를 포함하는 헤더
        - Entity headers

Body

    - optional
        GET, HEAD, DELETE, OPTIONS처럼 서버에 리소스 요청하면 필요 x
    - 필요한 경우
        POST, PUT
        단일-리소스 본문
        다중-리소스 본문


#### d. responses

    버전, 상태코드, 사유구절
    1.1 200 OK

Status line

    현재 프로토콜 버전, 상태 코드, 상태 코드에 대한 설명

Headers
    request와 같음

Body
    이하 동문

HTTP messages는 따로 정리 할 것 0808

### 4. HTTP 응답코드

    100 ~ : 정보전달

    200 ~ : 성공          
    200: OK, 정상            
    204: No Content, 보통 *특정내용을 삭제시* 해당 응답코드를 응답합니다.
    206: Range, 헤더를 지정한 요청을 응답할 때 사용합니다.

    300 ~ : Redirection, request완료를 위해 추가 동작이 필요합니다
    301: Moved Permanently, 영구적으로 URI 변경을 의미
    302: Found, 일시적인 URI 이동을 의미
    304: Not Modified, 변경 없음
    307: Temporary Redirect, 임시적인 redirect

    400 ~ : 클라이언트의 에러
    400: Bad Request, 잘못된 요청
    403: Forbidden, 접근권한없음
    404: Not Found, 요청 내용이 없거나 찾을 수 없음
    408: Request Timeout, 요청 타임아웃

    500 ~ : 서버의 에러
    500: Internal Server Error, 서버에러, 로직에러 발생시 자주 등장합니다.
    503: Service Unavailable, 서버 한계 초과등 오류
