### 1. 트랜잭션

> A는 인터넷으로 맥북을 샀다. 그런데 도중 오류로 결제가 이루어지지 않았고, 다시 주문하려다가 마음이 바뀐 A는 다시 주문하는 것을 포기했다. 그런데 3일뒤 맥북이 도착했다??

위 경우는 트랜잭션이 보장되지 않았기 때문에 일어난 행운이다. (판매자 입장에선 참사)

분석해보면 위의 사례는 크게 두가지의 쪼갤 수 없는 행동으로 구성되어있다.

    1. 결제  2.배송

하지만 생각해보자. 물리적으로는 두가지로 쪼개지지만 하나만 독단적으로 일어나면 어처구니없는 상황 아닌가?? 

즉 물리적으로는 나뉘어있지만 논리적으로는 하나의 작업으로 인식되어야하는 작업들인 것이다.

이것을 트랜잭션이라고 하며, 트랜잭션 하의 작업들은 모두 성공하거나 모두 실패해야하는 공동 운명체이다.

이 외에도 여러가지의 특징이 있는데 이를 ***ACID***라고 한다.

    1. A - Atomicity(원자성)
        모두 성공하거나 모두 실패하거나

    2. C - Consistency(일관성)
        짜여진대로(비즈니스 로직대로) 일관되게 수행되었는가

    3. I - Isolation(격리성)
        두개의 트랜잭션이 있다면 서로 영향을 끼치면 안된다.

    4. D - Durability(지속성)
        데이터베이스의 특징으로, 종료되어도 지속적으로 유지되어야 한다.

### 1-2 롤백, 커밋

> 앞선 A의 사례가 정상적으로 돌아가려면 어떤 일이 있어야 했을까??

1) 만약 결제와 배송이 성공적으로 진행되었다면, 이에 수반한 결과가 데이터에 정상적으로 변경되어 저장되어야 하며,

2) 하나라도 실패했다면, 모든 데이터들이 원래대로 돌아가야 할 것이다.

1번을 커밋, 2번을 롤백이라고 한다.

커밋을 하게되면 변경 결과가 데이터베이스에 영구 저장되며, 롤백을 하면 작업 직전 상태로 되돌아간다.

### 2. 스프링에서의 트랜잭션 구현

1. 코드베이스에서 구현한다.

영속성컨텍스트를 학습할때 보았던 EntityManagerFactory를 이용하면 EntityManager 인스턴스를 생성할 수 있는데, 이렇게 생성한 인스턴스로부터 트랜잭션 인스턴스를 생성할 수 있다.

```java
 private EntityManager em;
    private EntityTransaction tx;

    @Bean
    public CommandLineRunner testJpaBasicRunner(EntityManagerFactory emFactory) {
        this.em = emFactory.createEntityManager();
        
				// (1)
				this.tx = em.getTransaction();

        return args -> {
            tx.begin();
            .
            .
            .
            em.persist(member);
					
            tx.commit();    // (2)
        };
    }

```
이렇게 하는 것이 코드베이스이지 않나 싶다.. (찾아보는 중)

단점으로는 당연히 필수기능과 부가기능의 코드들이 섞여서 가독성이 악화되며 유지보수의 어려움을 낳는다는 것이 되겠다.

2. 선언형으로 구현한다.

클래스나 메서드 레벨에 @Transactional 을 추가해주기만 하면 되며, 둘 다 적용된 경우 메서드 레벨의 애너테이션이 우선적용된다.

3. AOP방식으로 구현한다.

선언형으로 작성하는 애너테이션마저 꼴보기 싫다면 AOP방식으로 적용 가능하다.

### 2-1 선언형에서의 여러 설정

1. Transaction Propagation

    Propagation.REQUIRED
        '트랜잭션 내놔'
        진행중인 트랜잭션에 참여한다. 없다면 새로 트랜잭션을 시작하고 참여한다.

    Propagation.REQUIRES_NEW
        '무조건 새거'
        트랜잭션을 하나 새로 만들어 진행한다. 이때, 진행중인 트랜잭션이 있다면 잠시 멈추게된다.

    Propagation.MANDATORY
        '트랜잭션 내놔, 근데 내가 안만들어'
        진행중인 트랜잭션에 참여한다. 없다면 예외를 발생시킨다.

    Propagation.*NOT_SUPPORTED*
        '난 혼자한다'
        트랜잭션은 필요없다. 해당 메서드가 실행될때는 진행중이던 트랜잭션은 잠시 멈춘다.

    Propagation.*NEVER*
        '트랜잭션 만들지마라'
        트랜잭션은 필요없다. 진행중인 트랜잭션이 있으면 예외를 발생시킨다.

2. Isolation Level

    Isolation.DEFAULT
        기본 값

    Isolation.READ_UNCOMMITTED
        다른 트랜잭션에서 커밋하지 않은 데이터를 읽는 것을 허용

    Isolation.READ_COMMITTED
        다른 트랜잭션에 의해 커밋된 데이터를 읽는 것을 허용

    Isolation.REPEATABLE_READ
        트랜잭션 내에서 한 번 조회한 데이터를 반복해서 조회해도 같은 데이터가 조회

    Isolation.SERIALIZABLE
        동일한 데이터에 대해서 동시에 두 개 이상의 트랜잭션이 수행되지 못하도록
